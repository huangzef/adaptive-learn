# Project Tracker - 项目实践追踪

基于计算参数自适应调整项目分解粒度的实战追踪技能。

## 核心设计原则

**参数驱动分解** — 项目的里程碑大小和颗粒度来自连续评分系统，不是固定值
**成果导向** — 每个子任务必须有可见的中间产出
**动态调整** — 根据检查频率参数调整进度检查节奏

---

## 连续评分系统集成

### 参数来源

在 PROJECT 阶段开始前，从会话文件读取：

| 参数 | 用途 |
|:-----|:-----|
| 颗粒度 | 子任务预估时长 |
| 里程碑大小 | 每个里程碑的子任务数量上限 |
| 检查频率 | 进度检查和确认的频率 |
| 理论实践比 | 调研/规划与编码/执行的比例 |

> **注意**：`练习密度` 参数在 PROJECT 阶段不使用。该参数用于 TEACHING 阶段控制练习题占比，而项目实践阶段以实际产出为导向，不涉及练习题设计。

### 项目类型适配

| 项目类型 | 里程碑大小调整 | 说明 |
|:---------|:-------------|:-----|
| 编程类 | × 1.0 | 基准 |
| 调研类 | × 0.8 | 子任务更细（需要更多文献/对比） |
| 设计类 | × 1.2 | 子任务更大（创意需要更大空间） |
| 理论类 | × 0.8 | 子任务更细（论证链条需要拆解） |
| 混合类 | × 1.0 | 平衡处理 |

## 第一阶段：项目解析

### 1.1 读取项目信息

从会话文件的 `## 项目信息` 分区读取：
- 项目目标
- 项目类型
- 已学习的里程碑（作为项目基础）

### 1.2 读取计算参数

从会话文件 frontmatter 读取：
- 颗粒度（子任务预估时长）
- 里程碑大小（每个里程碑的子任务数量）
- 检查频率（进度检查频率）
- 理论实践比（规划与执行比例）

### 1.3 分解子任务

根据项目类型和里程碑大小参数，将项目分解为子任务：

```
每个里程碑的子任务数量 = min(里程碑大小, 建议值)

项目总里程碑数 = ceil(项目复杂度评分 / 里程碑大小)
项目复杂度评分由以下因素估算：
- 项目目标数量（每个目标 = 1分）
- 技术难度（1-5分）
- 时间跨度（1-5分）
- 依赖复杂度（1-5分）
```

**子任务格式**：
```
子任务 N.[M]：{具体描述}
- 类型：{调研/设计/编码/测试/部署/文档}
- 预估时长：{基于颗粒度}分钟
- 成功标准：{可验证的产出物描述}
- 状态：待处理/进行中/已完成
```

### 1.4 应用 理论实践比

将子任务按 理论实践比 分配到"规划/调研"和"执行"两类：

```
规划/调研类子任务比例 = 理论实践比 / 100
执行类子任务比例 = 1 - (理论实践比 / 100)
```

在项目路线图中明确标注每个子任务的类型。

## 第二阶段：进度追踪

### 2.1 逐个子任务推进

每个子任务按以下流程推进：

```
1. 展示当前子任务（包含类型、预估时长、成功标准）
2. 引导用户完成
3. 检查频率触发时 → 进行进度确认（"这个子任务完成了吗？"）
4. 完成后 → 标记状态为已完成
5. 展示下一个子任务
```

### 2.2 进度检查机制

根据 检查频率 参数，设置检查节点：

```
检查频率 = N（每完成N个子任务后进行一次进度确认）
```

进度确认内容包括：
- 已完成子任务回顾
- 遇到的困难和解决方法
- 对下一个子任务的影响评估
- 必要时重新调整后续计划

### 2.3 进度可视化

```
项目进度：[██░░░░░░░░░░░░░░░░] 15%
当前里程碑：1 / 3
当前子任务：2 / 5
预估剩余时间：{基于未完成子任务数和颗粒度计算}分钟
```

### 2.4 异常处理

当用户在一个子任务上卡住超过预估时长（颗粒度）：

1. 分解子任务为更小的步骤
2. 提供渐进式帮助（L1提示 → L2拆解 → L3示范）
3. 如果仍无法完成，标记为"需要外部资源"并继续下一个

## 第三阶段：成果整合

### 3.1 子任务完成后整合

当所有子任务完成后，整合所有产出物：

```
## 成果清单
- [产出物1]: {描述} — {位置/链接}
- [产出物2]: {描述} — {位置/链接}
...
```

### 3.2 生成结课报告

基于以下输入生成叙事化结课报告：

- 项目目标 vs 实际完成情况
- 各里程碑的完成过程
- 遇到的挑战和解决方法
- 综合能力指数的变化轨迹
- 学习策略使用情况

### 3.3 Delta-Write：项目完成记录

将 PROJECT 阶段的完成记录追加到评分历史（delta-write）：

```markdown
| YYYY-MM-DD | 综合能力指数 | 基础能力 | 目标清晰度 | 时间资源 | 学习动机 | 项目完成 |
```

> **重要**：此技能只写入增量记录（delta），不直接修改完整参数快照。
> 写入后通知 learning-session-manager 触发快照刷新。

### 3.4 状态推进

写入最终状态：
```markdown
---
status: COMPLETED
---
```

## 参数驱动的自适应行为

| 参数 | 低值时的行为 | 高值时的行为 |
|:-----|:-----------|:-----------|
| 颗粒度 (15-120) | 短时快节奏子任务 | 长时间深度子任务 |
| 里程碑大小 (3-7) | 少量子任务，更大深度 | 多个子任务，更细粒度 |
| 检查频率 (1-5) | 较少检查，更多自主 | 频繁检查，更多确认 |
| 理论实践比 (20-60) | 偏执行，边做边调整 | 偏规划，先想清楚再做 |

## 与 learning-session-manager 的配合

### 触发时机

当 learning-session-manager 进入 **PROJECT 阶段**时，自动调用此技能。

### 数据读取

- `../learning_sessions/{session_id}.md` — 会话文件（包含参数）
- `../learning_profiles/{user_id}.md` — 学习画像

### 数据写入

| 分区 | 操作 | 说明 |
|:-----|:-----|:-----|
| `## 项目信息` | 读取 | 由 mentor-guide/socratic-tutor 写入 |
| `## 结课报告` | 写入 | 最终项目成果报告 |
| frontmatter | delta-write | 项目完成记录到评分历史 |

> **注意**：`## 教学复盘` 分区由 mentor-guide 写入，project-tracker 不写入该分区。

### 会话文件头部元数据记录

在会话文件头部元数据中记录：`当前技能：project-tracker，阶段：PROJECT`

## 快速参考

### 关键指令
| 指令 | 行为 |
|:-----|:-----|
| "开始项目" | 开始第一个子任务 |
| "继续下一子任务" | 推进下一个子任务 |
| "我需要帮助" | 诊断卡点，需要时调用 Mentor Guide |
| "暂停" | 保存进度 |
| "查看进度" | 展示当前进度摘要 |

### 项目类型对应
| 类型 | 子任务模板 |
|:-----|:-----------|
| 编程类 | 需求 → 选型 → 架构 → 实现 → 测试 → 部署 |
| 调研类 | 问题 → 收集 → 整理 → 分析 → 报告 → 修订 |
| 设计类 | 研究 → 概念 → 原型 → 评审 → 高保真 → 文档 |
| 理论类 | 定义 → 推导 → 验证 → 案例 → 总结 |
| 混合类 | 按需组合以上模板 |
