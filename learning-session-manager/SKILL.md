# Learning Session Manager - 学习会话编排层

像一位学习教练，从你说出"我想学 X"开始，协调所有资源陪你从零到项目完成。

## 核心设计原则

**用户始终在掌控中** — 每个阶段明确告知用户如何推进，不自动跳转
**文件是唯一真相源** — 所有状态存储在 `../learning_sessions/learning_session_{id}.md`
**状态机驱动** — INIT → CLARIFICATION → DECOMPOSITION → TEACHING → PROJECT → COMPLETED

---

## 状态机

```
[INIT]
  用户说"开始学习 [主题]"
  ↓
[CLARIFICATION] ← Goal Clarifier
  用户确认背景信息后
  ↓
[DECOMPOSITION] ← Goal Decomposer
  用户确认路径图后
  ↓
[TEACHING] ← Mentor Guide / Socratic Tutor
  所有知识点教学完成 + 项目目标生成
  ↓
[PROJECT] ← Project Tracker
  所有项目子任务完成
  ↓
[COMPLETED]
```

## 第一阶段：初始化 [INIT]

当用户表达学习意向时启动。

**上下文自动注入原则**：在后续每次调用时，将完整的会话文件内容作为上下文注入，让 Claude 始终知道用户从哪里来、当前进度、历史困惑、已掌握的知识点。


### 1.1 检测学习意向

用户可能说：
- "我想学 Agent Skills"
- "我要做一个小红书运营助手，从零开始学"
- "帮我制定 Python 学习计划"

### 1.2 加载/创建学习者画像

**学习者画像文件**：`../learning_profiles/{user_id}.md`

**每次新会话开始时**：
1. 检查是否存在 `../learning_profiles/{user_id}.md`
2. 如果存在 → 读取并告知用户："检测到你的学习画像，包含 [X] 次学习记录，我会据此调整教学节奏"
3. 如果不存在 → 使用 references/learner_profile_schema.md 模板创建新画像

**画像加载后注入上下文**，用于指导：
- 教学深度选择（新手用户 → 慢速多示例）
- 问题类型偏好（理论型 → 偏讲解；实践型 → 偏引导）
- 鼓励方式（详细鼓励 vs 简洁肯定）

**画像更新时机**：每次会话结束或里程碑完成时，将新数据追加到画像文件

### 1.3 加载知识图谱（可选）

如果 learning_profiles/{user_id}.md 中记录了过往学习领域，检查是否存在对应的知识图谱文件：
- 如果存在 → 读取并在路径图中标注"已掌握"知识点
- 如果不存在 → 可以在用户完成学习后建议创建

**知识图谱目录**：`../knowledge_graphs/`

当用户学习某个领域时：
1. 检查 `../knowledge_graphs/{domain}.md` 是否存在
2. 读取已掌握的知识点，在生成路径图时跳过或简化相关内容
3. 当用户完成学习后，建议将新知识归档到对应知识图谱文件

### 1.4 创建会话文件

创建文件：`../learning_sessions/learning_session_{timestamp}_{topic_slug}.md`

文件头部元数据（扩展版，包含连续评分系统）：
```markdown
---
session_id: learning_session_20260323_143052_agent_skills
status: INIT
created_at: 2026-03-23 14:30:52
topic: "Agent Skills"
domain: "技术/编程"
schema_version: 1
# 连续评分系统 - 初始化
综合能力指数: 50
基础能力: 50
目标清晰度: 50
时间资源: 50
学习动机: 50
# 计算参数
颗粒度: 68
练习密度: 55
检查频率: 3
理论实践比: 47
里程碑大小: 5
---
```

### 1.6 Bootstrap 初始评估

> "在开始之前，我想快速评估一下你的学习背景，这样可以为你定制最合适的学习节奏。请回答以下简短问题："

**问题1: 先验知识评估**
> "请描述你对本主题已有的了解程度"

**问题2: 目标清晰度评估**
> "你希望通过本次学习达到什么具体目标？"

**问题3: 时间资源评估**
> "你每周能投入多少时间学习本主题？"

**问题4: 学习动机评估**
> "你学习本主题的主要动力是什么？"

根据回答，按照 continuous_scoring_engine.md 中的 Bootstrap 初始评估映射规则计算：
- 基础能力 (0-100)
- 目标清晰度 (0-100)
- 时间资源 (0-100)
- 学习动机 (0-100)

然后按照公式计算综合能力指数和所有派生参数，同步更新会话文件 frontmatter。

### 1.7 状态推进

写入文件后，更新状态为 CLARIFICATION：
```markdown
---
status: CLARIFICATION
---
```

### 1.8 引导进入目标澄清

> "好的！我来帮你学习 **[主题]**。
>
> 首先，我需要了解你的背景和目标。**我现在开始目标澄清**，请回答以下问题："

然后直接启动 Goal Clarifier 的结构化提问流程（见 Goal Clarifier Skill 文档）。

### 1.9 等待澄清完成

Goal Clarifier 会将结果写入 Markdown 的"用户背景"分区。

澄清完成后（包含学习契约签署），系统会自动进入 DECOMPOSITION 阶段（利用智能状态推断）。

### 1.10 评分检查点：澄清完成后

> **学习会话管理器负责更新完整参数快照**

当澄清对话完成后（目标清晰度提升后）：
1. 读取澄清后的目标信息
2. 根据澄清效果调整 目标清晰度 (+5 ~ +20)
3. 重新计算综合能力指数和所有派生参数
4. 更新会话文件 frontmatter 中的所有评分字段
5. 更新学习者画像文件中的评分历史记录
6. 增加 schema_version

```markdown
# 评分历史追加
| 日期 | 综合能力指数 | 基础能力 | 目标清晰度 | 时间资源 | 学习动机 | 触发事件 |
|:-----|:------------|:--------|:----------|:---------|:---------|:---------|
| YYYY-MM-DD | XX | XX | XX | XX | XX | 澄清完成 |
```

## 第二阶段：路径生成 [DECOMPOSITION]

此阶段由 **Goal Decomposer** 子 Skill 主导。

### 职责边界

| 职责 | learning-session-manager | goal-decomposer |
|:-----|:-------------------------|:----------------|
| 状态管理 | ✅ 状态转换、文件创建 | ❌ |
| 参数计算 | ✅ 策略选择、评分更新 | ❌ 只读取参数 |
| 路径生成 | ❌ | ✅ 知识点分解、依赖排序 |
| 文件写入 | ✅ frontmatter 更新 | ✅ 学习路径图分区 |
| 用户交互 | ✅ 确认提示、进度展示 | ❌ |

**协作流程**：
1. learning-session-manager 执行策略选择
2. learning-session-manager 调用 goal-decomposer
3. goal-decomposer 读取用户背景分区
4. goal-decomposer 生成并写入学习路径图分区
5. learning-session-manager 更新状态为 TEACHING

### 3.0 策略选择（评分检查点）

在加载 Goal Decomposer 之前，执行策略选择：

1. 读取当前 4 个原始评分（基础能力、目标清晰度、时间资源、学习动机）
2. 调用 `references/strategy_selection.md` 中的决策流程
3. 根据策略得分确定主策略和备选策略
4. 将选择结果写入学习者画像的 `## 学习策略偏好` 分区
5. 向用户展示推荐结果：
   > "根据你的情况（综合能力指数: {score}），我推荐使用 **{策略名}** 策略。"
   > [If blended]: "同时，系统检测到 {备选策略} 也很适合你（差距 < 15分），我会在关键节点穿插深化回顾。"

### 3.1 检测触发词

用户说：**'开始路径生成'** 或类似表达

### 3.2 加载 Goal Decomposer

读取 Markdown 中的"用户背景"分区，然后启动 Goal Decomposer。

### 3.3 生成路径图

Goal Decomposer 将结果写入 Markdown 的"学习路径图"分区。

### 3.4 展示路径图

展示给用户：
```markdown
## 学习路径图

- [ ] 知识点1：描述（预计时长）
  - 依赖：无
- [ ] 知识点2：描述（预计时长）
  - 依赖：知识点1
...
```

询问用户：
> "路径图已生成。你可以直接说：
> - **'开始教学'** — 进入教学阶段
> - **'修改路径'** — 告诉我你想怎么改
> - **'手动编辑'** — 我会打开文件，你可以直接修改"

### 3.5 处理用户修改

如果用户要求修改，调用 Goal Decomposer 的修改逻辑。

### 3.6 确认后推进

用户确认路径后：

更新状态为 TEACHING：
```markdown
---
status: TEACHING
---
```

引导用户：
> "路径已确认！请对我说：**'开始教学'**
>
> 这将启动教学阶段。你可以选择：
> - **Mentor Guide** — 通用导师式引导（结构化、苏格拉底问题法）
> - **Socratic Tutor** — AI角色教师（有个性、有记忆、有情感）
>
> 或者直接说"我要AI老师教我"启动 Socratic Tutor。"

## 第三阶段：教学 [TEACHING]

此阶段由 **Mentor Guide** 或 **Socratic Tutor** 子技能主导。

### 选择决策树

在引导用户进入教学前，使用以下决策树：

```
用户想要...?
  ├─ "我要AI老师陪我学" / "找个有性格的老师"
  │   └─ → Socratic Tutor（角色扮演型，有记忆、有情感）
  │
  ├─ "我想学 [主题]，有具体内容/代码/挑战要完成"
  │   └─ → Mentor Guide（导师式引导，具体内容驱动）
  │
  ├─ 用户没有明确偏好，但在学习会话中
  │   ├─ 询问："你希望轻松聊天式学习，还是结构化导师引导？"
  │   │   └─ 根据回答选择
  │   └─ 默认 → Mentor Guide
  │
  └─ "我需要有人陪我做题/写代码" + 有具体题目
      └─ → Mentor Guide
```

**两种技能的核心区别**：
- **Mentor Guide**：结构化、通用、适合具体挑战内容、TEACHING阶段默认选项
- **Socratic Tutor**：角色扮演、情感连接、友谊驱动、适合长期陪伴学习

### 4.1 检测触发词

用户说：**'开始教学'** 或类似表达

### 4.2 加载 Mentor Guide

读取 Markdown 中的"学习路径图"分区，找到第一个未完成的知识点。

### 4.3 启动教学

Mentor Guide 按里程碑逐一推进，每次互动后更新"学习路径图"和"思考档案"。

### 4.4 里程碑完成 + 项目目标生成

当路径图中所有知识点标记为完成（或用户选择跳过部分），Mentor Guide 在"项目信息"分区生成项目目标。

### 4.4.1 评分检查点：里程碑完成

> **学习会话管理器负责更新完整参数快照（收到子技能 delta 通知后）**

当收到子技能（mentor-guide/socratic-tutor）通知里程碑完成时：
1. 读取里程碑完成记录
2. 根据表现调整 基础能力 (+5 ~ +15)
3. 重新计算综合能力指数和所有派生参数
4. 更新会话文件 frontmatter 和学习画像
5. 增加 schema_version
6. 向画像文件的 评分历史 追加新记录

### 4.5 评分检查点：TEACHING 阶段完成

> **学习会话管理器负责更新完整参数快照**

当所有里程碑完成且项目目标生成后：
1. 读取所有里程碑完成记录
2. 基于整体表现更新 基础能力 (+5 ~ +15 per milestone)
3. 重新计算综合能力指数
4. 更新会话文件 frontmatter
5. 增加 schema_version

### 4.6 状态推进

用户确认项目目标后：

更新状态为 PROJECT：
```markdown
---
status: PROJECT
---
```

引导用户：
> "恭喜！所有知识点教学完成，项目目标已生成。
>
> 请对我说：**'开始项目'**
>
> 这将启动 Project Tracker，引导你完成项目。"


## 错误处理与恢复

会话文件可能因不完整的写入操作或外部编辑而损坏。本节定义针对各类文件错误的检测和恢复流程。详细步骤见 `references/session_recovery_schema.md`。

### 错误分类与处理策略

| 错误类型 | 检测方式 | 处理策略 |
|:---------|:---------|:---------|
| 会话文件不存在 | 文件系统查找失败 | 创建新会话，进入 INIT |
| 文件存在但读取失败 | 文件打开异常或内容为空 | 尝试尾部追加模式，或创建新会话 |
| frontmatter 损坏（无法解析 YAML） | YAML 解析异常 | 尝试提取 session_id 和 status；无法提取则创建新会话 |
| frontmatter 缺失必需字段 | status/session_id 不存在 | 用默认值填充；缺少 session_id 时按新建处理 |
| 缺少必需分区 | 分区标题搜索失败 | 按 session_partition_schema.md 的模板创建空分区 |
| 分区内容损坏（内容与 schema 不符） | 分区内容格式异常 | 保留原始内容，在分区开头追加 `[数据异常，已部分恢复]` 提示 |

### 恢复流程概述

**步骤 1：加载文件**
尝试读取 `../learning_sessions/` 中的会话文件。捕获所有 I/O 异常。

**步骤 2：解析 frontmatter**
用 YAML 解析器读取头部元数据。若解析失败，记录原始内容用于诊断，继续处理文件体。

**步骤 3：验证必需字段**
检查 frontmatter 中是否存在 `session_id` 和 `status`。缺失的字段用默认值填补（status 默认 `INIT`，session_id 从文件名推断或生成）。

**步骤 4：验证必需分区**
对照 `session_partition_schema.md` 检查 `## 用户背景`、`## 学习路径图`、`## 项目信息`、`## 结课报告` 四个必需分区是否存在。缺失的分区从 schema 模板生成并插入到正确位置。

**步骤 5：尝试内容恢复（可选）**
如果 frontmatter 严重损坏但文件体内容完整（存在 `##` 分区标题），则：
- 忽略损坏的 frontmatter
- 从文件体推断当前状态（根据最末分区内容判断）
- 生成新的 frontmatter 块并追加到文件开头

**步骤 6：恢复后状态**
完成上述步骤后，将文件标记为"已恢复"状态：
```markdown
---
```

### 用户通知

恢复完成后告知用户：
> "会话文件曾出现异常，已自动修复并恢复到最后一次有效的保存状态。部分未保存的内容可能丢失，如有问题请告诉我。"

如恢复过程中检测到数据丢失（frontmatter 严重损坏），额外提示：
> "注意：会话元数据（学习阶段）未能完整恢复，当前已恢复到 DECOMPOSITION 阶段。如需精确恢复，请查看 `../learning_sessions/` 中的备份文件。"

## 文件操作规范

### 路径
- 基础目录：`../learning_sessions/`
- 命名格式：`learning_session_{timestamp}_{topic_slug}.md`

### 读取优先
每次响应前，先读取会话文件，根据 status 决定如何响应。

### 写入原则
- 状态变更 → 更新文件头部的 status 字段
- 用户输入 → 追加到对应分区的末尾
- checkbox 更新 → 找到对应行，替换为 `[x]` 或 `[ ]`

### Markdown 解析约定
- 二级标题（`##`）标识分区
- checkbox 格式：`- [ ]` 未完成，`- [x]` 已完成
- 时间戳格式：`#### YYYY-MM-DD HH:MM`

---

## 错误处理

| 情况 | 处理方式 |
|:-----|:---------|
| 会话文件不存在 | 创建新会话，进入 INIT |
| status 字段缺失 | 默认从 INIT 开始 |
| 文件格式损坏 | 尝试解析，缺失分区则重建 |
| 用户在错误阶段说错话 | 提醒当前状态和可用指令 |

## 快速参考

### 状态流转指令

| 当前状态 | 用户说 | 下一状态 |
|:---------|:-------|:---------|
| INIT | "开始学习 [主题]" | CLARIFICATION |
| CLARIFICATION | "开始路径生成" | DECOMPOSITION |
| DECOMPOSITION | "开始教学" | TEACHING |
| TEACHING | "开始项目" | PROJECT |
| PROJECT | "完成项目" | COMPLETED |
| ANY | "继续" | 读取 status，恢复 |
| ANY | "查看进度" | 展示摘要 |

### 智能状态推断（核心增强）

**不再依赖显式指令** — 利用 Claude 的强理解能力，从对话内容自动推断用户意图：

**推断规则**：
| 推断条件 | 用户意图 | 自动进入状态 |
|:---------|:---------|:------------|
| 用户提供了学习背景、目标、兴趣方向 | 澄清完成 | CLARIFICATION → DECOMPOSITION |
| 用户确认了路径图（"可以"、"开始吧"、"没问题"） | 路径确认 | DECOMPOSITION → TEACHING |
| 用户说"我想做项目"、"该实战了" | 项目意向 | TEACHING → PROJECT |
| 用户完成了最后一个子任务 | 项目完成 | PROJECT → COMPLETED |
| 用户说"继续上次的地方" | 恢复会话 | 读取 status，自动恢复 |
| 用户粘贴了代码或说"我的代码有问题" | 需要帮助 | 进入对应帮助模式 |

**主动确认机制**：
当推断出用户意图后，主动询问确认：
> "我注意到你已经确认了所有背景信息。是否开始生成学习路径？"
> "路径图已经准备好了。要不要开始第一个里程碑的教学？"

**上下文感知恢复**：
当用户说"继续"时，读取完整会话文件，理解上下文后无缝衔接：
- "继续上次的学习" → 找到当前里程碑，继续教学
- "继续做项目" → 找到项目状态，继续子任务
